# Checkpoint — 2026-05-01_2350

## Context Summary

Single-session hotfix. v2.11.1 shipped with a confirmed read-side bug in the
robots.txt REST endpoints: POST wrote the physical file and the DB option
correctly, but GET /robots-txt returned `source: "wordpress_default"` and
`content: ""` immediately after a successful POST. GET /status also fired
a false-positive `physical_robots_txt_bypass` warning for managed files.

## Accomplishments

1. **v2.11.2 shipped** — robots.txt read-side stale-state bug fixed and
   released. Both fix commit (`da87d9d`) and zip commit (`02163f8`) pushed
   to main.

2. **Root cause isolated** — POST was already writing `rrseo_robots_txt`
   option correctly. Failure was WP persistent object cache (common on
   LiteSpeed hosts with Redis/file-cache) returning the pre-write value on
   subsequent GET requests. PHP `file_exists` stat cache was a secondary
   contributor.

3. **Status bypass warning corrected** — `physical_robots_txt_bypass` now
   suppressed when `rrseo_robots_txt` option is non-empty (plugin is managing
   the file). Previously fired on every physical robots.txt regardless of
   ownership.
