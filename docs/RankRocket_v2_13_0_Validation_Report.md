## Side-findings (not gap-related, surfaced during testing)

### Side-finding 1 — Title has a leading space after RankMath removal
**Where:** Homepage in State 2 and State 3
**Observed:** `<title> SEO Agency for Service Businesses | Rank Rocket Co</title>` (note leading space)
**State 1 value:** `<title>SEO Agency for Service Businesses | Rank Rocket Co</title>` (no leading space)
**Suspect:** when RankRocket assembled the title from its stored postmeta, a separator or prefix template added an extra leading space. Cosmetic only but worth tracking down.

### Side-finding 2 — Meta description has a double space
**Observed in State 2/3:** `"Rank Rocket Co, is the leading  SEO Agency..."` (double space before SEO)
**Same source likely as Side-finding 1.**

### Side-finding 3 — RankMath does not wipe postmeta on delete
**Observation:** After deleting both Rank Math SEO and Rank Math SEO PRO plugins entirely from disk, the `migrate-legacy` dry-run still returned 3 posts with migratable data. The `rank_math_*` postmeta keys are still in `wp_postmeta`. The lone migrated value across 3 posts is `focus_keyword: '%title%'` — the same broken template token pattern observed on Salvo Metal Works.
**Why:** RankMath has a "Remove Data on Uninstall" toggle in General Settings that is OFF by default. Plugin deletion only removes plugin files, not the data it wrote to wp_postmeta and wp_options.
**Implication:** For sites where the cut-over from RankMath to RankRocket is intended to be permanent, the operator should either (a) enable "Remove Data on Uninstall" in RankMath BEFORE deleting, or (b) run a separate `DELETE FROM wp_postmeta WHERE meta_key LIKE 'rank_math%';` after the plugin is gone. Without this cleanup, the legacy postmeta will linger indefinitely — not harmful (RankRocket reads from `rr_seo_*` keys) but storage bloat over time.

### Side-finding 4 — /llms.txt rebuild verification not possible programmatically
Because G-12 is not implemented, there is no deterministic way to trigger an llms.txt regeneration. The file does regenerate on its own (verified during the Salvo audit) but with no `/regenerate` endpoint, operators can't confirm the new state without waiting for an indeterminate cache cycle.

### Side-finding 5 — The `url:` pattern returns 422
This is actually the **best behavior** of any unimplemented routing pattern in v2.13.0. The validator correctly identifies `url:` as invalid input and rejects with HTTP 422. The `term:` and `tax:` patterns silently accept and never fire, which is worse — operators can't tell their snippets are dead without manually viewing rendered HTML. **Recommendation:** apply the same 422 rejection to `term:` and `tax:` until G-01 actually ships, so silent failures become loud failures.

## Recommended next steps for the plugin

1. **Ship G-01 in v2.13.1** — the routing logic is the headline feature of the version bump and it's not actually present. Either ship it or revert to v2.12.x and rename the version label.
2. **Fix G-02** — at minimum, return HTTP 422 with `{"error":"term writes not supported in this version, use /terms/<id> when available"}` when a term ID is passed to `/preview-update` or `/update`. Today the plugin says `valid: true` and silently no-ops, which is dangerous.
3. **Ship G-03, G-08, G-11 (single), G-12** in v2.13.1 or v2.14.0 — all are small surface-area additions.
4. **Tighten validation on POST /snippets** — reject any `display_on` value that isn't in the actual allowlist (matching what's actually emitted), with the same 422 behavior `url:` already gets. This stops silent failures across the board.
5. **Investigate the leading-space / double-space cosmetic issues** in title/description assembly after RankMath deactivation (Side-findings 1 and 2). Suspect a separator concatenation bug.

## Recommended mu-plugin enhancement

The current `RRC_SEO_TAX_META_DESC` module only fires for `product_cat`. Generalize to fire for any taxonomy term that has a description set:

```php
// In mu-plugin, replace the current condition:
if ( ! is_tax( 'product_cat' ) && ! is_product_category() ) {
    return;
}
// With:
if ( ! is_tax() && ! is_category() && ! is_tag()
     && ! ( function_exists( 'is_product_category' ) && is_product_category() ) ) {
    return;
}
```

This would let the mu-plugin emit meta descriptions on WordPress core `category` and `post_tag` archives too, not just WooCommerce `product_cat`. Useful for content sites like rankrocket.co. The same change should be applied to the OG/Twitter Card module.

This was the reason R-13 returned `meta_description_count=0` on rankrocket.co — the site uses `category` archives, not `product_cat`, so the mu-plugin had nothing to do. After this generalization the mu-plugin would emit meta description on `/category/uncategorized/` and equivalents.
